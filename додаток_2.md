# ДОДАТОК 2 — Майно йде за користувачем; користувачів не видаляємо, а блокуємо (з гардом)

**Дата:** 2026-06-16
**Версія:** (правки до ТЗ v2.2)
**Пріоритет:** Високий
**Статус:** Завершено ✅

---

## 1. ЗАВДАННЯ
1. Прибрати **каскадне видалення майна** з системи повністю. Майно **йдеться за користувачем** (утримувачем).
2. Користувачів з системи **не видаляти** — лише **блокувати** (деактивація).
3. **Гард блокування:** не давати блокувати користувача, поки на ньому є **активне** майно (статус `active` або `expired` — тобто не `written_off` і не `transferred`). Після списання/передачі всього майна — блокувати можна.
4. DELETE-ендпоінт користувача — **заборонити**.

---

## 2. ПІДСТАВА (логіка)
Списане/передане майно лишає людину в історії (акти, CRUD-журнал). Тому:
- блокування (а не видалення) зберігає запис про людину;
- активне майно «тримає» користувача, доки воно не списане/не передане;
- видалення структурних сутностей (область/підрозділ) **не руйнує** активи — вони лишаються за утримувачем.

---

## 3. РЕАЛІЗАЦІЯ

### 3.1. Схема БД
- `assets.department_id`: **`ON DELETE CASCADE` → `ON DELETE SET NULL`**.
  - Наслідок: видалення підрозділу (або області через `region→department CASCADE`) **не видаляє** активи — їхня прив’язка до підрозділу обнуляється (`department_id = NULL`), а `responsible_user_id` лишається («йдуть за користувачем»).
- `region → department CASCADE` — **залишається** (це структура; активи все одно виживають).
- `assets.responsible_user_id ... ON DELETE RESTRICT` — лишається (БД-рівневий захист історії; реально користувачів не видаляємо).

### 3.2. Бекенд (`routes/users.js`)
- `PATCH /api/users/:id/toggle-active` — **гард**:
  - при спробі **блокування** (is_active: true → false) рахуємо активи:
    `SELECT COUNT(*) FROM assets WHERE responsible_user_id=$1 AND status NOT IN ('written_off','transferred')`;
  - якщо > 0 → **403** з повідомленням «Неможливо заблокувати: на користувачі N активних одиниць майна. Спишіть або передайте їх» (`active_assets` в тілі).
  - розблокування (false → true) — без обмежень.
- `DELETE /api/users/:id` — **завжди 403**: «Видалення користувачів заборонено. Замість цього заблокуйте користувача.»

### 3.3. Фронтенд (`pages/UsersPage.jsx`)
- Прибрано кнопку «Видалити користувача» (Popconfirm).
- Додано кнопку **блокування/розблокування** (LockOutlined / UnlockOutlined) → `usersAPI.toggleActive`. Помилка гард показується повідомленням.

## 4. ТЕСТ (пройдено)
1. Створити користувача `editor_test` (editor, dept 1).
2. Адмін вносить актив із `responsible_user_id = editor_test` → актив активний.
3. `PATCH /users/6/toggle-active` → **403** («1 активна одиниця»).
4. Списати цей актив (act write-off).
5. `PATCH /users/6/toggle-active` → **200** (заблоковано).
6. `DELETE /users/6` → **403** (видалення заборонене).
7. `assets.department_id` після потенційного видалення dept → **SET NULL**, актив виживає, `responsible_user_id` збережено.

---

## 5. ВИДАЛЕННЯ ЧЕРЕЗ БД (зауваження, на потім)
Технічно `DELETE FROM users WHERE user_id=…` сирим SQL можливе, **АЛЕ** `assets.responsible_user_id ON DELETE RESTRICT` заборонить це, поки хоч один актив посилається на людину. Отже:
- застосунок видалення не пропонує (тільки блокування);
- БД-рівень сам захищає історію від розриву;
- видалити можна лише юзера без жодного активу (або знявши обмеження — адмін-дія поза застосунком).
